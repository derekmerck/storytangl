Vue 3 + Vuetify 3 reference component guide

An implementation guide — not a mockup — for the StoryTangl web client. Every component is designed against the canonical RuntimeEnvelope fragment stream, not the legacy block adapter. Props, emits, and payloads are derived only from the v1.5 contract (VOCAB_DIGEST.md §2–§7); no backend field, fragment type, or service shape is invented here. Components are written <script setup lang="ts">, strict mode, Composition API.

Scope discipline. Tier S components are specified to build now. P1/P2 widgets (piece, zone, roll, interpretation) are named but kept design-only — see §6. Visual styling that doesn't touch a contract is isolated there too.

noteOn the framework choiceread before adopting Vuetify

Vuetify is Material Design: rounded surfaces, elevation shadows, Roboto, ripple. The StoryTangl brand is the opposite — square engineering-notebook surfaces, hairline rules, no shadows, the Newsreader / JetBrains Mono / Inter Tight stack. Vuetify can wear the brand, but only after a theme + defaults override (given in §6). If the team isn't committed to it, a headless base would reach brand fidelity faster:

  • Reka UI (Radix Vue successor) or PrimeVue unstyled — own every pixel, a11y primitives handled. Best brand fit.
  • PrimeVue styled — strongest DataTable for the table value_type.
  • Vuetify 3 — batteries included; budget time to override Material defaults.

This guide targets Vuetify 3 as requested. Crucially, the contract surface is framework-agnostic: the component names, props, emits, and payloads below hold whether the primitive underneath is a v-card or a Reka Primitive. Swapping frameworks later changes §1's "Vuetify primitive" line and §6's theme — nothing else.

Treat the Vuetify mapping as the rendering layer; treat props/emits/payloads as the contract boundary. Keep the dispatch (FragmentRenderer) and the store (useEnvelope) free of any Vuetify import so the floor stays portable.

§0Contract types & the render pipelineauthoritative · do not extend

These TypeScript types are a transcription of the wire contract — the source of truth is the engine, not this file. The renderer dispatches on fragment_type through a registry; an unknown type degrades to StUnknownFragment (§4) rather than throwing. The store applies control mutations and never re-renders silently-updated state as prose.

contract · fragments
interface RuntimeEnvelope {
  cursor_id: string
  step: number
  fragments: Fragment[]
  metadata?: EnvelopeMetadata
}

type Fragment =
  | ContentFragment    // content
  | AttributedFragment // attributed
  | MediaFragment      // media
  | GroupFragment      // group
  | KvFragment         // kv
  | ChoiceFragment     // choice
  | ControlFragment    // control
  | UserEventFragment  // user_event

interface FragmentBase {
  uid: string
  fragment_type: string
  visibility?: "public" | "owner_only" | string[]
}

commit bus
One event type bubbles to the host: commit with { edgeId: string; payload: CommitPayload }. Only choice-family components emit it; everything else is presentational or store-bound.

§2Commit inputs · accepts.kindStCommitInput · gate 4

StCommitInput dispatches on accepts.kind (closed set of 7, digest §4) and emits the typed payload up through StChoice. Every richer modality has a typed/click fallback in the same turn. pieces and place reference P2 piece/zone data — the inputs are specified now but only exercised once those fragments are present.

§3ProjectedState · the status railStStatusRail · 5 value_types

StStatusRail renders projected_state.sections[]. Each section dispatches on value.value_type (digest §5). section.kind is a free label rendered as the heading; the client does not branch on it. All five sub-components are presentational (no emits) — the rail reflects state, it doesn't commit.

No-color & high-contrast

  • Severity never by hue alone — glyph (✓ ! ‼) + sr-only label + border accompany every emphasis color.
  • Theme exposes a high-contrast variant; forced-colors media query swaps to system colors and keeps borders.
  • Locked choices read [LOCKED] in text, not a faded tint; disabled state has a non-color cue.
  • Focus rings are 2px and never removed; :focus-visible throughout.
  • Hit targets ≥ 44px; prefers-reduced-motion kills skeleton shimmer + ripple.

Long labels & overflow

  • Choice text wraps to 2 lines then ellipsizes with full text in title / tooltip.
  • KvRow keys truncate with text-overflow; value stays visible (the value is the data).
  • table value_type → horizontal scroll + sticky first column; never squashes columns illegibly.
  • Speaker names + badges truncate at a max-width; tags wrap to a new line, never overflow the rail.
  • All measures use ch/%, not fixed px, so zoom to 200% reflows cleanly.
